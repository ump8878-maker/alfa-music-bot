# Анализ рейтинга чатов: предвзятость и целевая модель

## 1. Проблемы в текущей архитектуре

### 1.1 Предвзятость в пользу «редких» жанров

- **`app/services/matching.py` → `calculate_rarity_score`**  
  Явно задаётся список «мейнстрим» жанров (`pop`, `hiphop`, `rock`) и набор «популярных» артистов. Чем больше таких жанров/артистов в профиле, тем **ниже** rarity (0 = мейнстрим, 1 = редкий). Жанры techno, indie, house, jazz и т.д. по умолчанию считаются «более редкими» и получают более высокий rarity.

- **`app/rating.py` → `compute_user_taste_score`**  
  Использует `profile.rarity_score` как бонус: `rarity_bonus = int(profile.rarity_score * 25)`. В итоге пользователь с поп/хип-хоп/рок получает меньший «вкус», а чат с такими участниками — меньший рейтинг. Рейтинг чатов и пользователей фактически ранжирует по «редкости» вкуса, а не по активности и качеству данных.

- **Цепочка влияния**  
  Редкость (в matching) → сохраняется в `MusicProfile.rarity_score` → используется в `compute_user_taste_score` → попадает в `calculate_chat_rating` через средний score. Иерархия «крутости» жанров заложена в формулу рейтинга.

### 1.2 Что сейчас не учитывается в рейтинге

- **Активность участников** — нет явного учёта (есть только факт «прошёл квиз»).
- **Полнота прохождения квиза** — не различаем «заполнил всё» и «минимум полей».
- **Согласованность профиля чата** — нет метрики «насколько вкусы в чате совпадают между собой».
- **Вовлечённость** — доля прошедших квиз от размера чата не используется (плюс `member_count` может быть не заполнен).

## 2. Модули для создания/переработки

| Модуль | Действие |
|--------|----------|
| **`app/rating.py`** | Переписать `compute_user_taste_score`: убрать использование `rarity_score`; ввести «полноту профиля» (заполненность полей квиза) и «разнообразие» (число жанров/артистов без учёта их «редкости»). |
| **`app/chat_rating.py`** | Переписать `calculate_chat_rating`: не использовать `compute_user_taste_score` в текущем виде как «качество вкуса»; формула только из: число участников (активность), полнота данных, согласованность (внутренние матчи), разнообразие жанров (без штрафа за поп/рок и т.д.), при возможности — вовлечённость. |
| **`app/services/matching.py`** | **Не менять** `calculate_rarity_score`: оставить для матчинга и отображения «редкость» в профиле. Использовать только там; в рейтинге не использовать. |

## 3. Что лучше не трогать

- **`app/services/matching.py`** — логика матчинга и `calculate_rarity_score`: оставить как есть; рейтинг не должен зависеть от rarity.
- **`app/models/music_profile.py`** — поле `rarity_score` и `profile_type`: оставить для отображения и матчинга.
- **Отображение редкости** в профиле (profile_card, card_generator, inline) — оставить как метрику «интерес/редкость», не как «качество».
- **Handlers, БД, миграции** — без изменений структуры; меняются только формулы в `rating.py` и `chat_rating.py`.
- **Сканер чата** (`chat_analytics`) — не влияет на рейтинг, не трогать.

---

## 4. Целевая модель рейтинга (без иерархии жанров)

- **Активность:** число участников, прошедших квиз (кап, чтобы не раздувать рейтинг огромными чатами).
- **Полнота квиза:** средняя по чату «полнота» профиля: жанры + артисты + mood + listening_time (все поля равноправны, без весов по жанрам).
- **Согласованность:** средний match_score между участниками чата (насколько вкусы в чате согласованы).
- **Разнообразие:** число уникальных жанров в чате; бонус за разнообразие без штрафа за «массовые» жанры.
- **Вовлечённость:** при наличии `member_count` — доля прошедших квиз; иначе не учитывать.

Итог: одна понятная формула, без использования `rarity_score` и без предзаданной иерархии качества жанров.

## 5. Внесённые изменения (итог)

- **`app/rating.py`**: `compute_user_taste_score` переписан — только полнота квиза и разнообразие (число жанров/артистов), без `rarity_score`. Добавлена `profile_completeness(profile)` для использования в рейтинге чата.
- **`app/chat_rating.py`**: `calculate_chat_rating` переписан — пять компонент (активность, полнота, согласованность, разнообразие, вовлечённость), без использования «редкости» и без иерархии жанров.
- **`app/services/matching.py`**: в докстринг `calculate_rarity_score` добавлено предупреждение не использовать эту метрику для рейтинга.
